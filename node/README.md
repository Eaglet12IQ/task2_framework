## Контракт модуля

Каждый модуль должен экспортировать объект со следующей структурой:

```javascript
export default {
  name: "ModuleName",           // Уникальное имя модуля
  requires: ["OtherModule"],    // Список зависимостей
  register(container) {         // Регистрация служб в контейнере
    container.addSingleton("service.key", factory);
    container.addTransient("service.key", factory);
  },
  async init(container) {       // Инициализация модуля
    // Логика инициализации
  }
};
```

## Архитектура

### Ядро (не зависит от модулей)
- **container.js** — контейнер внедрения зависимостей с поддержкой:
  - `addSingleton(key, factory)` — один экземпляр на приложение
  - `addTransient(key, factory)` — новый экземпляр каждый раз
  - `get(key)` — получение службы
  - `getMany(prefix)` — получение всех служб с префиксом

- **moduleLoader.js** — загрузчик модулей:
  - `loadModulesFromConfig()` — загрузка модулей из JSON-конфигурации
  - `buildOrder()` — топологическая сортировка с проверкой циклов

- **errors.js** — кастомные ошибки

### Модули (подключаются без изменения ядра)
1. **Core** — базовые службы (clock, storage)
2. **Logging** — журналирование событий
3. **Validation** — проверка данных
4. **Export** — экспорт в файл (зависит от Validation)
5. **Report** — формирование отчётов (зависит от Export)

## Проверки

### Тесты (13 тестов)
1. Порядок запуска с простой цепочкой зависимостей
2. Параллельные ветви зависимостей
3. Отсутствующий модуль в конфигурации
4. Недостающая зависимость
5. Большая цепочка зависимостей
6. Сообщение об отсутствующем модуле
7. Обнаружение циклической зависимости
8. Синглтон возвращает тот же экземпляр
9. Transient создаёт новый экземпляр каждый раз
10. Зависимости внедряются через контейнер
11. Модуль не может создать несуществующий сервис
12. Модуль с несовместимой версией вызывает ошибку
13. Синглтон и Transient имеют разное время жизни

### Демонстрационные сценарии
- **basic** — работает, порядок: Core → Logging
- **full** — работает, порядок: Core → Logging → Validation → Export → Report
- **missing** — ошибка с понятным сообщением
- **cycle** — ошибка с понятным сообщением

## Почему ядро не знает о модулях

Ядро работает только с абстрактным контрактом:
1. Читает список файлов из JSON-конфигурации
2. Импортирует каждый как ES-модуль
3. Ожидает поле `name`, `requires`, `register`, `init`
4. Вызывает методы в нужном порядке

Для добавления нового модуля нужно:
1. Создать файл в `modules/` или `external-modules/`
2. Добавить имя файла в `config/modules.json`
3. Никаких изменений в ядре не требуется

## Дополнительные возможности

### Загрузка внешних модулей
Модули загружаются автоматически из папки `external-modules/`. Это позволяет расширять приложение без изменения основной директории модулей.

### Проверка совместимости версий
Каждый модуль должен указывать версию в поле `version`. Ядро проверяет:
- Наличие обязательных полей контракта (name, requires, register, init)
- Совместимость версий (модули с большей мажорной версией отклоняются)

### Два времени жизни объектов
- **Singleton** — один экземпляр на всё приложение
- **Transient** — новый экземпляр при каждом запросе
